RPC Endpoints

Dwellir provides managed, production-grade JSON-RPC endpoints for Astar Network, Shiden (Kusama canary), and Shibuya (testnet). All endpoints require API key authentication and support both HTTPS (request/response) and WebSocket (persistent connection with subscriptions). Endpoints are load-balanced across geographically distributed infrastructure for low latency and high availability.

HTTPS vs WebSocket

HTTPS (Request / Response)

  • Best for stateless queries: eth_call, eth_getBalance
  • Easy to use with standard HTTP clients and serverless functions
  • Batch multiple calls in one POST request using JSON-RPC batch arrays
  • Connection overhead per request; suitable for low-to-medium traffic

WebSocket (Persistent)

  • Required for subscriptions: eth_subscribe, chain_subscribeNewHeads
  • Lower latency for frequent queries — reuse single connection
  • Maintain one persistent connection per process, multiplex requests
  • Implement reconnection with exponential backoff for resilience

API Key Authentication

Include your API key as the last path segment of the endpoint URL. No additional headers are required. The API key is tied to your Dwellir account and used for rate limit tracking and usage analytics.

# HTTPS — curl example
curl -X POST   -H "Content-Type: application/json"   -d '{"jsonrpc":"2.0","method":"eth_blockNumber","params":[],"id":1}'   https://api-astar.n.dwellir.com/YOUR_API_KEY

# WebSocket — wscat example
wscat -c wss://api-astar.n.dwellir.com/YOUR_API_KEY
> {"jsonrpc":"2.0","method":"chain_getHeader","params":[],"id":1}

Best Practices

  • Batch Requests

    Group multiple read calls into a single JSON-RPC batch array to reduce round-trip overhead. Ideal for fetching multiple balances or block data simultaneously. Most Ethereum libraries support batching natively.

  • Cache Metadata

    Substrate runtime metadata (state_getMetadata) changes only on runtime upgrades. Cache it locally and refresh only after detecting a spec version change via state_getRuntimeVersion. This dramatically reduces repeated metadata fetches.

  • Exponential Backoff

    Implement retry logic with exponential backoff for transient 429 (rate limit) or 5xx errors. Start at 1 second, double each retry, cap at 60 seconds. Add jitter to prevent thundering-herd scenarios.

  • WebSocket Reconnection

    WebSocket connections can drop due to network events. Implement automatic reconnection that re-establishes subscriptions and re-syncs missed blocks using chain_getBlockHash to fill any gaps.
